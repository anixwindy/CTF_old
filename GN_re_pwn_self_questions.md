# RE -> Pwn Self Review Questions

用途：

- 自我審核
- 類似面試題
- 用來判斷你現在在哪個階段
- 用來判斷你能不能往下一階段走

搭配方式：

- 這份不是答案本
- 這份是「問題清單」
- 最好先口頭回答，再去查
- 如果你連問題都看不懂，通常代表這一層還沒到

---

## 1. 使用規則

每一題先用這個標準給自己分數：

- `0` = 完全不會
- `1` = 聽過名詞，但講不清楚
- `2` = 能用自己的話解釋
- `3` = 能解釋，還能自己做最小驗證

建議通關條件：

- 每一階段平均至少 `2`
- 該階段的「硬問題」至少大部分能答
- 至少有幾題你真的能動手驗證，不是只有嘴上會講

你要特別注意：

- `知道名詞` 不等於 `真的會`
- `看過 writeup` 不等於 `自己能做`
- `payload 能跑` 不等於 `你理解為什麼`

## 2. Phase 0：環境、工具、工作流

這一層不是技術高潮，但沒有它，後面節奏會爛掉。

### 基本問題

1. `file` 是在看什麼？
2. `strings` 能給你什麼線索？不能給你什麼？
3. `xxd` 或 `hexdump` 什麼時候要看？
4. `objdump` 和 `readelf` 大致差在哪？
5. `gdb` 是做什麼的？它不只是「程式停下來」而已，還能幹嘛？
6. `Ghidra` 是拿來看什麼？它不是萬能在哪裡？
7. 為什麼拿到 binary 不應該第一步就直接執行？
8. 對一個未知 binary，你的固定起手流程是什麼？

### 怎麼做題

1. 你怎麼確認一個檔案是不是 ELF？
2. 你怎麼快速看一個 binary 裡面有沒有可疑字串？
3. 你怎麼看一個 binary 的 section 或 symbol 資訊？
4. 你怎麼在 `gdb` 啟動一個程式？
5. 你怎麼在 Ghidra 找到 `main` 或主要函式？

### 硬問題

1. 為什麼「先觀察，再動態」通常比「先跑跑看」更穩？
2. 如果一個 binary 幾乎沒有明顯字串，你下一步會改看什麼？

### 可以往下的標準

- 你能自己對一個小 binary 做最基本的資訊蒐集
- 你有固定流程，不是每次都亂試

## 3. Phase 1：`C`、記憶體、Linux 基礎

這一層是 `RE` 和 `Pwn` 的共同地板。

### 是什麼

1. pointer 和 array 關係是什麼？哪裡像，哪裡不一樣？
2. `char buf[16]` 在 stack 上大概代表什麼？
3. string 為什麼要有 `\\0`？
4. signed / unsigned 差在哪？
5. cast 為什麼可能讓值變掉？
6. overflow 是什麼？它在 C 裡為什麼危險？
7. `struct` padding 是什麼？
8. `malloc` / `free` 在概念上做了什麼？
9. `memcpy`、`strcpy`、`printf` 哪些地方常出問題？
10. local variable 為什麼常跟 stack frame 一起被講？

### 怎麼做

1. 你怎麼用一個小 C 程式驗證 signed / unsigned 比較？
2. 你怎麼驗證一個字串在記憶體裡最後有 `\\0`？
3. 你怎麼觀察一個 local buffer 在 stack 上的大概位置？
4. 你怎麼判斷一個 copy 操作有沒有長度風險？
5. 你怎麼解釋 `printf(user_input)` 為什麼危險？

### 判斷題

1. 為什麼很多 `Pwn` 題在本質上都能回到「C 對記憶體做了危險的事」？
2. 如果一段程式一直在做 `buf[i]`、`buf[i+1]` 這種操作，你第一個會留意什麼？

### 硬問題

1. 為什麼 pointer 算術跟型別大小有關？
2. 為什麼 signed / unsigned 混在一起比較，常讓人判斷錯誤？
3. 為什麼一個小小的 off-by-one 也可能很危險？

### 可以往下的標準

- 你能用自己的話講清楚 pointer、string、stack、overflow
- 你能看出明顯危險函式與長度風險

## 4. Phase 2：`x86-64`、ABI、stack frame、`gdb`

這是你從「看 C」走到「看 binary」的橋。

### 是什麼

1. `mov`、`lea`、`cmp`、`test` 各自在幹嘛？
2. `call` 和 `ret` 在控制流上代表什麼？
3. 條件跳轉和 `cmp/test` 的關係是什麼？
4. `rsp` 和 `rbp` 大致在做什麼？
5. stack frame 是什麼？
6. `System V AMD64 ABI` 大概在規定什麼？
7. 函式參數常放在哪些 register？
8. 回傳值通常放在哪？
9. return address 在哪裡？
10. 為什麼 `lea` 常不只是「取地址」？

### 怎麼做

1. 你怎麼在 `gdb` 看 register？
2. 你怎麼在 `gdb` 看 stack？
3. 你怎麼下 breakpoint？
4. 你怎麼單步執行？
5. 你怎麼在關鍵 `cmp` 前停住？
6. 你怎麼觀察某個輸入值最後跑到哪裡？

### 判斷題

1. 看到 `[rbp-0x20]`，你第一反應應該是什麼？
2. 一個函式結尾前把某個值放進 `rax`，這通常代表什麼？
3. 如果某個條件跳轉沒走成功分支，你最應該往回看什麼？

### 硬問題

1. 為什麼 `cmp` 不是真的「把東西存起來」，但後面的跳轉卻依賴它？
2. 為什麼你不能只背指令意思，還要看資料流？
3. 你怎麼從一小段組語反推它大概原本是什麼 C 邏輯？

### 可以往下的標準

- 你能看懂小函式的流程
- 你能指出參數、局部變數、比較點、回傳點
- 你能用 `gdb` 驗證一個簡單猜測

## 5. Phase 3：`RE` 靜態分析

這一層的目標不是會很多工具，而是會抓線索。

### 是什麼

1. 靜態分析是什麼？
2. 反編譯和反組譯差在哪？
3. 為什麼不能完全相信反編譯結果？
4. symbol、import、string、constant 各能提供什麼線索？
5. magic value 是什麼？
6. jump table 大概代表什麼控制流型態？
7. 為什麼很多 RE 題喜歡用 XOR、ADD、ROL 這些簡單變換？

### 怎麼做

1. 你怎麼找輸入點？
2. 你怎麼找比較點？
3. 你怎麼找成功 / 失敗分支？
4. 你怎麼看出某段像不像字串比對？
5. 你怎麼看出某段像不像 table lookup？
6. 你怎麼從反編譯跳回關鍵組語確認？

### 判斷題

1. 如果題目裡有很多固定常數，你第一個會懷疑什麼？
2. 如果一段驗證被拆成很多小函式，這通常是在增加什麼難度？
3. 如果輸入不是直接比對，而是先被轉換，你要先找哪兩個點？

### 硬問題

1. 為什麼 RE 不能只問「答案是什麼」，而要問「它怎麼算出來」？
2. 你如何區分「直接比較」和「先變換後比較」？
3. 如果反編譯看起來很醜，你為什麼應該回去看組語？

### 可以往下的標準

- 你能對小 crackme 做基本靜態推理
- 你能說出輸入大概經過哪些轉換

## 6. Phase 4：`RE` 動態分析與驗證

這一層是從「我猜」變成「我證明」。

### 是什麼

1. 動態分析和靜態分析在目的上差在哪？
2. breakpoint、watchpoint、大致差在哪？
3. register、stack、memory 三者在追資料時各看什麼？
4. 為什麼很多時候應該在比較點前後停住？
5. 為什麼動態分析不是拿來亂試密碼？

### 怎麼做

1. 你怎麼在驗證函式入口停住？
2. 你怎麼追某個 byte 被轉換後的值？
3. 你怎麼確認某個常數真的參與比較？
4. 你怎麼在 `gdb` 看某段記憶體是 string 還是 raw bytes？
5. 你怎麼知道你現在看的是原始輸入、轉換中資料，還是最終比較值？

### 判斷題

1. 如果你靜態看出像 XOR，但不確定 key，動態時最應該停哪裡？
2. 如果程式檢查很多段輸入，你怎麼決定先驗證哪一段？
3. 如果某條路徑一直走失敗分支，你會回頭補看哪類資訊？

### 硬問題

1. 為什麼動態分析應該服務你的假設，而不是取代你的思考？
2. 你怎麼證明「這個 byte 是從輸入第幾個位置來的」？
3. 如果反編譯和執行結果對不上，你要先相信什麼？為什麼？

### 可以往下的標準

- 你能用 `gdb` 驗證自己的 RE 假設
- 你能說出資料流，不只是最後答案

## 7. Phase 5：`ELF`、連結、保護機制

這一層是 `RE` 接 `Pwn` 的關鍵橋段。

### 是什麼

1. `ELF` 是什麼？
2. section 和 segment 差在哪個層面？
3. symbol 是什麼？
4. relocation 大概在解決什麼問題？
5. `PLT` / `GOT` 是什麼？
6. 動態連結是什麼？
7. libc 在 userland 程式裡扮演什麼角色？
8. `NX`、`Canary`、`PIE`、`ASLR`、`RELRO` 各在防什麼？

### 怎麼做

1. 你怎麼看一個 binary 的保護機制？
2. 你怎麼辨識某次函式呼叫是不是經過 `PLT`？
3. 你怎麼看一個 symbol 或 relocation？
4. 你怎麼用自己的話解釋為什麼 leak 一個位址有價值？

### 判斷題

1. 如果開了 `NX`，你對攻擊方向的第一個想法應該變成什麼？
2. 如果有 `PIE` 和 `ASLR`，為什麼硬編碼位址通常不穩？
3. 為什麼 `Canary` 讓單純的 stack smash 變麻煩？

### 硬問題

1. 為什麼 `Pwn` 題在真正利用前，先看保護機制很重要？
2. 為什麼 `PLT/GOT` 會常出現在 `ret2libc` 或 leak 類思路裡？
3. 你怎麼用簡單語言解釋動態連結，不要只背名詞？

### 可以往下的標準

- 你能看懂保護組合
- 你能說出它們對利用方向的影響

---

## 8. Phase 6：`Pwn` 第一階段，stack overflow 與 `ret2win`

這是 `Pwn` 的第一個真正入口。

### 是什麼

1. stack overflow 是什麼？
2. local buffer、saved `rbp`、return address 的相對關係是什麼？
3. offset 是什麼？
4. 為什麼覆蓋 return address 能改控制流？
5. `ret2win` 是什麼類型的題？

### 怎麼做

1. 你怎麼確認程式有沒有 crash？
2. 你怎麼找 offset？
3. 你怎麼確認 `RIP` 是否被控制？
4. 你怎麼找目標函式位址？
5. 你怎麼解釋一個最小 `ret2win` payload 的結構？

### 判斷題

1. 如果某個 buffer 明顯比輸入短，你第一個懷疑什麼？
2. 如果 return address 沒被蓋到，你應該回頭檢查什麼？
3. 如果有 `Canary`，你的原本想法為什麼可能失敗？

### 硬問題

1. 為什麼 `Pwn` 不是「看到 overflow 就直接亂塞地址」？
2. 你怎麼證明成功點是控制流被改，而不是純粹 crash？
3. 為什麼 stack layout 在這一層是核心，而不是附屬知識？

### 可以往下的標準

- 你能自己找 offset
- 你能解釋 payload 每段的作用
- 你能說出保護機制怎麼影響這題

## 9. Phase 7：`ROP`、leak、`ret2libc`

這一層開始不能只靠最直線的 stack smash。

### 是什麼

1. `ROP` 是什麼？
2. gadget 是什麼？
3. 為什麼 `NX` 會逼你常常改走 `ROP`？
4. leak 是什麼？
5. libc base 是什麼？
6. `ret2libc` 的核心想法是什麼？

### 怎麼做

1. 你怎麼找一個可用 gadget？
2. 你怎麼看一個 leaked address 大概屬於哪裡？
3. 你怎麼從 leak 推回 base？
4. 你怎麼把函式參數放到正確位置？
5. 你怎麼解釋一條 `ROP chain` 每一段在幹嘛？

### 判斷題

1. 如果位址不固定，為什麼你常常需要先 leak？
2. 如果你已經知道 libc base，下一步通常能做什麼？
3. 如果某題不能直接執行 shellcode，你應該開始想哪一路？

### 硬問題

1. 為什麼 `ROP` 不是單純把很多地址串起來而已？
2. 為什麼你一定要理解 calling convention，`ROP` 才不會變成背模板？
3. 為什麼 leak 在很多題裡是中間目標，不是最終目標？

### 可以往下的標準

- 你能講清楚 leak -> base -> call 這條鏈
- 你能說明每個 gadget 的用途

## 10. Phase 8：format string

這一層會大幅強化你對 stack 與讀寫 primitive 的理解。

### 是什麼

1. format string bug 是什麼？
2. 為什麼 `printf(user_input)` 可能導致任意讀？
3. 為什麼 `%n` 類型可能導致任意寫？
4. 為什麼 format string 題常能 leak stack、libc、pie？

### 怎麼做

1. 你怎麼判斷一題 format string 主要是拿 leak 還是拿 write？
2. 你怎麼解釋「參數位置」在 format string 題裡的意義？
3. 你怎麼驗證某個輸出對應的是 stack 上的哪個值？

### 判斷題

1. 如果題目能印出你控制的格式字串，第一個該懷疑什麼？
2. 如果能 leak 位址，為什麼這通常很快會接到 `Pwn` 主線？

### 硬問題

1. 為什麼 format string 題不是只背 `%p`、`%n` 就算會？
2. 為什麼它本質上還是在處理資料讀取位置與寫入效果？

### 可以往下的標準

- 你能分辨 leak 題與 write 題
- 你能解釋 stack 上參數位置的意義

## 11. Phase 9：heap 入門

這層不要太早進，但也不能永遠逃。

### 是什麼

1. heap 和 stack 最大差異是什麼？
2. `malloc` / `free` 在 allocator 角度大概做了什麼？
3. chunk 是什麼？
4. `tcache` 大概是什麼？
5. double free 是什麼？
6. use-after-free 是什麼？
7. off-by-one 在 heap 題裡為什麼可能危險？

### 怎麼做

1. 你怎麼觀察一個程式裡 chunk 的配置與釋放順序？
2. 你怎麼判斷 freed chunk 為什麼又被重用？
3. 你怎麼用最簡單方式解釋 heap 題中的「控制了什麼」？

### 判斷題

1. 如果一個指標 free 後還被使用，你第一個想到什麼？
2. 如果同一塊記憶體被重複 free，為什麼危險？
3. 為什麼 heap 初學時不該先背一大堆版本技巧？

### 硬問題

1. 為什麼 heap 題最重要的是看懂資料結構與流向？
2. 為什麼 heap 題常需要更穩的 leak 與任意寫觀念？

### 可以往下的標準

- 你看得懂最小 heap 題的大致 chunk 流向
- 你能解釋 UAF、double free、off-by-one 的風險

## 12. Crypto 生存區

這不是你的主線，但你至少要不會連入口都看不懂。

### 是什麼

1. Base16 / Base32 / Base64 大概長什麼樣？
2. XOR 是什麼？
3. Caesar / ROT 是什麼？
4. Vigenere 是什麼？
5. substitution cipher 是什麼？
6. hash 和 encryption 差在哪？
7. RSA 在 CTF beginner 題裡通常會出現哪些概念詞？

### 怎麼做

1. 你怎麼判斷一串資料像不像 Base64？
2. 你怎麼驗證某題是不是單 byte XOR？
3. 你怎麼判斷一題是不是簡單替換或位移？

### 判斷題

1. 什麼時候應該先試編碼/解碼，而不是直接往高深理論想？
2. 為什麼對你的主線來說，Crypto 應該維持生存能力就好？

### 可以往下的標準

- 你能快速辨識最常見 beginner crypto 題型
- 你知道什麼時候該先寫一個小腳本驗證

## 13. 最重要的跨階段問題

這些問題如果一直答不好，通常代表主線還沒有真的打通。

1. 你能不能把一段 binary 的行為講成清楚的人話？
2. 你能不能指出輸入點、比較點、成功點、失敗點？
3. 你能不能分清楚「我猜的」和「我驗證過的」？
4. 你能不能說出這題最重要的 primitive 是什麼？
5. 你能不能解釋保護機制如何改變利用方向？
6. 你能不能說出某個 payload 為什麼長那樣？
7. 你能不能在不看 writeup 的情況下，先做最小推理？

## 14. 自我判定表

### 你大概還在 `C / 組語前期`

如果你常卡在：

- pointer / array / string 講不清楚
- 看不懂 stack frame
- `cmp`、`test`、條件跳轉常常混亂

### 你大概到了 `RE 初階`

如果你已經能：

- 靜態找線索
- 找出輸入點與比較點
- 用 `gdb` 驗證少量假設

但還常卡在：

- 多函式資料流
- 轉換後比較
- 反編譯和真實組語對不上

### 你大概可以進 `Pwn`

如果你已經能：

- 畫出粗略 stack layout
- 解釋保護機制
- 說出 overflow 怎麼改控制流

### 你大概在 `Pwn 初階`

如果你已經能：

- 找 offset
- 做最小 `ret2win`
- 理解為什麼要 leak

但還常卡在：

- `ROP` 鏈設計
- libc base 推回
- format string / heap 的資料模型

---

## 15. 最後的使用建議

不要一次把全部題目做完。

比較好的用法：

1. 每兩週只審一個 phase
2. 先口頭答
3. 再挑 `3` 到 `5` 題做最小驗證
4. 記錄哪些題是「不會」、哪些題是「會說但不會做」

真正有價值的不是：

- 我看起來知道很多名詞

而是：

- 我能不能把這層知識講清楚，並自己做最小驗證
